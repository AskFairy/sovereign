## 1. 这段代码本身很快

`querySelector('h1')`、取 tags，都是毫秒级。正常情况下整段应在 **10ms 以内** 跑完。

## 2. 真正拖慢的是这几件事

### ① `document.body.innerText` 太重

面试鸭题目页 DOM 很大：答案区 + 1900+ 条评论 + 大量 React 节点。

`innerText` 会：
- 遍历整棵 DOM 树
- 触发 layout/reflow
- 序列化所有可见文本

在 MCP 远程执行时，这一步最容易卡住。你之前那次 **~167 秒超时**，很像卡在这里或 DevTools 协议层，而不是 JS 算得慢。

### ② Chrome DevTools MCP 有协议开销

每次 `evaluate_script` 的路径是：

```text
Cursor → MCP → Chrome DevTools Protocol → 页面 JS → 回传结果
```

页面复杂或浏览器忙时，协议层会排队/超时。你之前还出现过：

- `Runtime.evaluate timed out`
- `Execution context was destroyed`（页面导航/刷新）

说明 **MCP 连接不稳定**，不是脚本写错。

### ③ 页面本身有前端问题

之前控制台有：
- React error #418 / #423
- `MaxListenersExceededWarning: 11 SHOW_LOGIN_MODAL listeners`

页面 JS 异常时，DevTools 注入脚本更容易挂住。

## 3. 应该怎么改（更快）

**不要读 `document.body.innerText`，缩小范围：**

```js
() => {
  const clean = (s) => (s || "").replace(/\s+/g, " ").trim();
  const main = document.querySelector("main main") || document.querySelector("main");
  const title = clean(document.querySelector("h1")?.textContent);
  const tags = [...main.querySelectorAll('a[href*="/tag/"]')].map((a) => clean(a.textContent));
  const headerText = clean(main?.querySelector("h1")?.parentElement?.textContent || "");
  const difficulty = /简单|中等|困难/.exec(headerText)?.[0] || "";
  return { title, tags, difficulty, url: location.href };
}
```

**更稳的做法（Codex 那套）：**

| 方式 | 速度 | 说明 |
|------|------|------|
| 本地 Playwright 脚本 | 最快 | 直连浏览器，无 MCP 中转 |
| `take_snapshot` 解析 a11y 树 | 较快 | 你之前 snapshot **几秒内**就回来了 |
| 分步小脚本 | 中等 | 一次只取 title，一次只取 answer panel |
| 整页 `body.innerText` | 最慢 | 应避免 |

慢的原因可以概括成：

> **在超重的 SPA 页面上，通过 MCP 远程执行 `document.body.innerText`，协议 + DOM 遍历双重放大延迟。**

| 策略 | 建议 |
|------|------|

### 6.4 错误处理

| 场景 | 处理 |
|------|------|
| 登录失效 | 检测「未登录」→ 暂停，提示重新登录 |
| 跳离题目页 | `pathname` 校验，失败记 failed |
| 某 tab 为空 | 写占位 + 记 warn，不阻断其他 tab |
| 图片下载失败 | 保留远程 URL，记 log |
| 网络超时 | 重试 3 次 |

为什么抓取的内容里有：点击登录查看完整内容？

**AI最终回复**
**有登录态，但站点对正文做了截断**，所以抓到的 DOM 里会带上「点击登录查看完整内容」。

## 原因

面试鸭在页面上**只渲染一段预览**，后面用「… 点击登录查看完整内容」挡住剩余部分。当前脚本用 `innerText` 整段读 `section`，这段提示会一起被读进来。

常见触发情况：

| 情况 | 表现 |
|------|------|
| **未登录 / 登录过期** | 正文被截断，末尾出现该文案 |
| **已登录但非 VIP** | 部分题 `needVip=1`，仍只给预览 |
| **未展开完整答案** | 面经类要点「隐藏答案」才出全文；普通题若未展开，也可能只有预览 |

面经类（从「隐藏答案」后切片）往往较完整；**普通面试题**（从「回答重点」切片）更容易只看到预览 + 这句提示。

## 证据

- `汇总v1.0/AI 应用架构设计面试题.md`：多题末尾是 `... 点击登录查看完整内容`
- 同目录下 `AI 应用架构设计面试题.md`（旧版）：同一道医疗题曾有**完整**长文 → 说明**同一账号、不同抓法/会话**结果可以差很多
- `汇总v1.0/Java 基础面试题.md`：大量题目都有该文案

## 和「有登录态」的关系

启动时登录检查能通过（页面不显示「未登录」），但：

1. **headless 会话里的 cookie 可能不完整**，或 VIP 权益未带上  
2. 脚本**没有检测**正文里是否含「点击登录查看完整内容」，仍按 `answerLen > 80` 判成功  
3. 未对普通题做「展开全文」（类似面经的「隐藏答案」）或走 API 拉完整答案

## 建议改法（给 Codex / 后续实现）

1. **抓后校验**：正文含「点击登录查看完整内容」→ 标为 `partial` 或失败，不要当完整成功  
2. **抓前确认**：用 API `user/get/login` 看 `userRole` 是否为 `fvip`，并确认 `hasVipAuth`  
3. **找完整内容来源**：Network 里是否有 `best_answer` 等返回全文的接口  
4. **DOM 展开**：普通题也尝试点「隐藏答案」/展开后再读 `innerText`  
5. **重新登录**：`node open_browser.mjs` 用 VIP 账号登录后再抓

需要的话我可以改脚本：自动检测该文案、尝试展开答案，并把含截断的题记入 `partialIds` 方便补抓。

脚本**没有**账号密码自动登录，靠的是：

1. 你手动登录一次（`open_browser.mjs`）
2. Cookie 保存在 `.browser-profile/`
3. 后续抓取复用这个 profile
