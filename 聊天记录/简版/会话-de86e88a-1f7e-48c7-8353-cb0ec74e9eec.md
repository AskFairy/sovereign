## 1. 这段代码本身很快

`querySelector('h1')`、取 tags，都是毫秒级。正常情况下整段应在 **10ms 以内** 跑完。

## 2. 真正拖慢的是这几件事

### ① `document.body.innerText` 太重

面试鸭题目页 DOM 很大：答案区 + 1900+ 条评论 + 大量 React 节点。

`innerText` 会：
- 遍历整棵 DOM 树
- 触发 layout/reflow
- 序列化所有可见文本

在 MCP 远程执行时，这一步最容易卡住。你之前那次 **~167 秒超时**，很像卡在这里或 DevTools 协议层，而不是 JS 算得慢。

### ② Chrome DevTools MCP 有协议开销

每次 `evaluate_script` 的路径是：

```text
Cursor → MCP → Chrome DevTools Protocol → 页面 JS → 回传结果
```

页面复杂或浏览器忙时，协议层会排队/超时。你之前还出现过：

- `Runtime.evaluate timed out`
- `Execution context was destroyed`（页面导航/刷新）

说明 **MCP 连接不稳定**，不是脚本写错。

### ③ 页面本身有前端问题

之前控制台有：
- React error #418 / #423
- `MaxListenersExceededWarning: 11 SHOW_LOGIN_MODAL listeners`

页面 JS 异常时，DevTools 注入脚本更容易挂住。

## 3. 应该怎么改（更快）

**不要读 `document.body.innerText`，缩小范围：**

```js
() => {
  const clean = (s) => (s || "").replace(/\s+/g, " ").trim();
  const main = document.querySelector("main main") || document.querySelector("main");
  const title = clean(document.querySelector("h1")?.textContent);
  const tags = [...main.querySelectorAll('a[href*="/tag/"]')].map((a) => clean(a.textContent));
  const headerText = clean(main?.querySelector("h1")?.parentElement?.textContent || "");
  const difficulty = /简单|中等|困难/.exec(headerText)?.[0] || "";
  return { title, tags, difficulty, url: location.href };
}
```

**更稳的做法（Codex 那套）：**

| 方式 | 速度 | 说明 |
|------|------|------|
| 本地 Playwright 脚本 | 最快 | 直连浏览器，无 MCP 中转 |
| `take_snapshot` 解析 a11y 树 | 较快 | 你之前 snapshot **几秒内**就回来了 |
| 分步小脚本 | 中等 | 一次只取 title，一次只取 answer panel |
| 整页 `body.innerText` | 最慢 | 应避免 |

## 4. 结论

慢的原因可以概括成：

> **在超重的 SPA 页面上，通过 MCP 远程执行 `document.body.innerText`，协议 + DOM 遍历双重放大延迟。**
